---
title: 规划与拆解
description: 把大任务拆成可审查、可验证的小步，让 agent 稳定执行。
---

agent 擅长执行，但面对庞大、开放式的任务时容易跑偏。一个好的规划，能把模糊的需求转化为一串小而可验证的步骤 —— 而这恰恰是 agent 最拿手的。

## 先规划，再动手

对任何不那么琐碎的改动，先要一份规划，并在写任何代码_之前_审查它：

```markdown
"写代码之前，先列出你的方案：你会改哪些文件、按什么顺序、每一步如何验证。
等我确认后再动手。"
```

审查一份规划成本很低，审查 400 行写错的代码则不然。许多 agent（包括 Claude Code）都有专门的 plan 模式，只做调研和提案，不碰文件。

## 一份好规划包含什么

- **将要改动的文件**，以及每处大致改什么
- **一个顺序** —— 依赖在前，让每一步都建立在稳固的基础上
- **每个单元一个验证步骤** —— 要跑的测试、类型检查、或手动检查
- **明确的未知项** —— "我不确定会话是怎么存的，我会先看 `auth/`"

如果一份规划缺少验证步骤，那是个危险信号 —— 打回去重写。

## 沿着自然接缝拆解

在各部分能够独立审查、独立测试的地方切分工作：

```markdown
功能："用户可以把自己的数据导出为 CSV"

1. 加上 exportUserData(userId) 服务 + 单元测试     （纯逻辑，易验证）
2. 加上 GET /api/export 接口 + 集成测试            （接线）
3. 加上 UI 里的"导出"按钮和下载流程                （展示层）
```

每一步都是独立的提交、独立的审查。如果第 2 步错了，你也没把它和 UI 工作纠缠在一起。

## 每一步小到能被验证

当你能盯着一步的 diff、在一两分钟内_确信_它对不对时，这一步的粒度就合适。如果做不到，那就是太大了 —— 让 agent 进一步拆分。

## 现实变化时就重新规划

规划是廉价且可丢弃的。如果 agent 中途发现数据模型和假设不符，那就停下来重新规划，而不是任由它带着一个错误假设即兴发挥。花三十秒重新规划，好过顺着错误方向调试一个小时。
