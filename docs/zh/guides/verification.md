---
title: 验证循环
description: 用测试、类型和运行时检查，让 agent 自证其工作。
---

agentic coding 的超能力不在于 agent 会写代码 —— 而在于它能_检查自己的代码并修复它_，全程不需要你介入。但前提是你得给它一条验证的途径。没有反馈信号的 agent，只是在自信地瞎猜。

## 给 agent 一条"知道自己错了"的途径

在要求改动之前，先确保有一个 agent 能读到的客观信号：

- **测试** —— `npm test`、`pytest`、`go test`
- **类型检查** —— `tsc --noEmit`、`mypy`
- **linter** —— `eslint`、`ruff`
- **构建** —— 能不能编译通过？
- **应用运行** —— 页面能加载吗？接口返回 200 吗？

然后告诉它去用：

```markdown
"实现这个改动，然后运行 `npm test` 和 `npm run typecheck`。
修好所有失败项，两个都通过之前不要停。"
```

## 验证循环

一个健康的 agentic loop，会收敛到一个真实的信号上：

1. **实现**改动
2. **运行**检查（测试 / 类型 / 构建）
3. **读取**真实输出 —— 而不是对输出的猜测
4. **修复**并重复，直到全绿

要警惕的失败模式：agent _声称_"现在应该能过了"，却什么都没运行。坚持让它真的去跑。"应该能过"不等于"过了"。

## 测试是最好的信号

类型和 linter 抓的是形状错误，测试抓的是行为错误。只要可能，就优先用一个失败的测试作为规格 —— 它精确地告诉 agent"完成"意味着什么，并让它无人值守地迭代。这正是 [TDD](/zh/workflows/tdd) 的核心思想。

```markdown
"这是 cart.test.ts 里一个描述折扣规则的失败测试。
在不改动测试的前提下让它通过。"
```

## 不要轻信，要验证 —— 也包括你自己

agent 的验证不能取代你的验证，它只是把你的验证前置了。当 agent 报告全绿时，你仍然要审查 diff。但此时你审查的代码已经能编译、能过测试、能过类型检查 —— 于是你的注意力就能放在_设计与意图_上，而非机械性的 bug。

## 当没有信号时

有些工作没有自动检查 —— 一处视觉微调、一句文案改动。那就把循环显式化：让 agent 说清楚该看哪里，然后你去看。同时也想想这个缺口值不值得补上 —— 一个快照测试或冒烟测试，往往立刻就能收回成本。
