---
title: 上下文管理
description: 会话变长时，让 agent 保持专注与准确。
---

agent 的表现，取决于它上下文窗口里的内容。太少，它就靠猜；太多，它就抓不住重点。管理好上下文，是 agentic coding 中杠杆率最高的技能之一。

## 上下文为什么会退化

会话变长时，会发生三件事：

- **信号被稀释。** 40 条消息之前那句重要指令，要和此后每次工具调用产生的噪声竞争注意力。
- **过时事实残留。** agent 早期读过的文件可能已经变了，它却仍按旧版本推理。
- **成本与延迟上升。** 每一轮都要重新处理整个窗口。

## 新任务，开新会话

最有效的一个习惯：**一个任务，一个会话。** 做完一个功能、要转到无关的另一个时，清空上下文，而不是继续堆叠。

```markdown
✅ 做完认证功能 → /clear → 开始计费功能
❌ 用一个 3 小时的会话横跨六个互不相关的任务
```

在 Claude Code 里，`/clear` 会重置对话，同时保留你的项目配置（`CLAUDE.md`、settings）。如果你想继续但甩掉累积的噪声，用 `/compact` —— 它会把目前的历史总结一下再继续。

## 把持久事实放进 CLAUDE.md

任何 agent 在_每个_任务里都需要的东西，都应放进 `CLAUDE.md`，而不是放在需要你反复重复的聊天消息里：技术栈、约定、关键命令、架构边界。这能让你的提示保持简短，且在 `/clear` 后依然有效。结构见 [CLAUDE.md](/zh/reference/claude-md)。

## 只拉入刚好够用的上下文

当某个任务需要特定上下文时，精准地把 agent 指向它，而不是一股脑全倒进去：

```markdown
✅ "读一下 src/payments/stripe.ts 和 src/types.ts 里的 Payment 类型，然后加上退款支持。"
❌ "这是我整个代码库，加上退款支持。"
```

点名相关文件，用最小的噪声引入最对的上下文 —— 也顺便教会 agent 下次该去哪儿找。

## 警惕漂移

上下文已经过时的迹象：

- agent 引用了一个不再存在的函数或文件
- 它再次建议一个你已经否决过的改动
- 它的回答越来越含糊，或开始和先前自相矛盾

一旦发现漂移，用 `/compact` 或 `/clear`，再用一句清晰的当前目标重新锚定。
