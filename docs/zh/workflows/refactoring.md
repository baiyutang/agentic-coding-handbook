---
title: 大重构
description: 安全地穿过庞大、机械的改动，而不至于一次性搞坏一切。
---

大重构既是 agent 大放异彩之处，也是它破坏力最大之处。让 agent 能在 200 个文件里重命名一个符号的那种机械触达力，同样能让它一次搞坏 200 个文件。诀窍在于结构：绝不一次改完所有，并且始终保持一个绿色信号。

## 先建好安全网

在动任何东西之前，先确保你能察觉到什么时候出了问题：

```markdown
"重构之前，先跑一遍完整测试套件和类型检查，好让我们有一个已知为绿的基线。
把结果报给我。"
```

如果你要改的区域覆盖率很薄，先补上_特征测试_（characterization tests）—— 钉住当前行为的测试。这样重构才有东西可保护。

## 以可审查的切片推进

抵制"一击定音"的巨型 diff。把重构拆成一个个都能保持构建为绿的步骤：

```markdown
重构：把零散的 db.query() 调用替换为新的 Repository 层。

1. 引入 Repository 接口 + 一个实现（先不改任何调用方）
2. 迁移模块 A 使用它；保持测试为绿
3. 迁移模块 B；保持测试为绿
4. ...
5. 当没有任何东西再引用旧路径时，移除旧的 db.query()
```

每个切片都是一个你能独立审查、必要时能独立回退的提交。

## 优先做机械、可验证的变换

对于有清晰不变量的变换 —— "行为完全一致，只有结构改变" —— agent 最可靠。把这点明说出来：

```markdown
"这是一次纯重构 —— 没有行为改动。如果某个测试需要改动才能通过，
就停下来告诉我，而不是去改它。"
```

最后那句是护栏：一次"纯重构"中若有测试被改动，那是行为发生了偏移的信号，不该被糊弄过去。

## 让循环保持紧凑

每个切片之后都跑一遍套件，而不是拖到最后。一个在第 7 个切片才崩的重构，在前 6 个切片都为绿的情况下很容易定位；而一个 15 文件的 diff 一次性失败，就是一场调试苦战。这就是[验证循环](/zh/guides/verification)在大规模下的应用。

## 知道何时停下重新规划

如果某个切片开始膨胀 —— "这次重命名牵动了配置，配置又牵动了启动，然后……" —— 那是代码库在告诉你[规划](/zh/guides/planning)需要调整了。停下来，重新界定范围，再继续。别让 agent 顶着不断扩大的爆炸半径硬推。
