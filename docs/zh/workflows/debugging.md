# 调试

当 agent 被指向一个**可复现的失败**、并被要求找根因时，它是强力的调试者。当你对它说"把这个报错弄没就行"时，它是危险的调试者 —— 你会得到一个掩盖症状的补丁，或一串每修一个就坏另一个的死循环。

## 从复现开始

你能交给 agent 的最好的东西，是一种按需触发 bug 的方式：

- 一个捕捉到它的失败测试（最理想 —— 它同时充当修复的[验证](/zh/guides/verification)）
- 能复现的确切命令和输入
- 真实的报错输出 —— 堆栈、日志 —— 而不是你的转述

```markdown
"这是一个复现 bug 的失败测试，外加堆栈。找到根因，解释它，然后修复。不要改测试。"
```

## 坚持先诊断再修复

要防的失败模式：agent 猜一个修法，不行，再猜一个。逼它先诊断。

```markdown
"在改任何代码之前：解释到底发生了什么、为什么，并指出负责的具体行号。然后再提出修法。"
```

一个你能核验的根因陈述，永远胜过一个默默打上的补丁。如果解释站不住脚，你就在错误修复落地之前抓住了它。

## 警惕「修一个坏一个」死循环

一个经典的 agent 陷阱：它修好 A，结果坏了 B；修好 B，又坏了 A；如此循环。你正陷在里面的迹象：

- 同几个文件被来回编辑
- 测试 X 变绿时测试 Y 变红，然后又反过来
- 每次"修复"都是局部补丁，而非对底层原因的改动

看到这个，就**停下、拉远视角**。这个循环几乎总意味着真正的问题在上一层 —— 一个错误的假设、一个糟糕的抽象、或两个真正冲突的需求。

```markdown
"别再打补丁了。这两个修法一直在互相抵消。退一步：是什么共同的假设或设计决策让两者同时失败？修那个，而不是症状。"
```

## 控制爆炸半径

- 用能解决问题的最小改动去修根因 —— 抵制"既然来了顺手改"
- 修完跑**完整**套件，而不只是那一个测试 —— 确认没有别的地方回归
- 把复现测试留在代码库里，让这个 bug 无法悄悄回来

## 当 agent 卡住时

如果两三次尝试都不收敛，通常是缺上下文，而非缺能力。给它更多信号 —— 相关日志、周边模块、数据实际怎么流动 —— 或[升级到更强的模型](/zh/stack/models)来做诊断。原因清楚后，再让更快的模型去应用修复。
