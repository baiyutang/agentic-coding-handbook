# 多 agent 与并行

从"不错"跃升到"卓越"的生产力，往往不是靠更好的单会话 —— 而是靠**并行地跑好几个**。Anthropic 2026 趋势报告点出了从单 agent 到协同团队、从分钟级任务到工作数小时的 agent 这一转变。这一页讲的是如何做到这点而不制造混乱。

## 为什么要并行

一个 agent 在一个上下文窗口里顺序地做完一个任务。多个 agent，各有各的上下文，可以同时攻不同的独立部分 —— 而且每个都更锐利，因为它的上下文只限于自己那一小块。[10x → 20x 的跃升](/zh/stack/going-10x)在很大程度上就是这个：多个已验证的任务同时在跑，而非只有一个。

## 两种模式

### 1. 并行的独立任务

最简单也最可靠：把工作拆成**不碰同一批文件**的部分，各跑一个 agent。

- 功能 A 在一个会话，不相关的 bugfix B 在另一个
- 一个 agent 写测试，另一个写文档
- 每个各自成一个分支、各自[审查](/zh/workflows/code-review)

让这一切保持理智的规则：**沿着没有共享状态的接缝并行。** 两个 agent 同时编辑同一个模块，就是一场等着发生的合并冲突。

### 2. 编排者 + 专家

对更大的任务，一个"编排者"拆解工作，把限定范围的子任务分给专门的 agent（架构、实现、测试、审查），再整合结果。有些工具通过 subagent 直接支持；你也可以手动做 —— 跑一个规划会话，再为每一步派生聚焦的会话。

- **强项：** 跨独立上下文窗口的并行推理；每个专家保持专注
- **代价：** 你现在要协调好几份产出，并干净地整合它们

## 让它跑通需要什么

并行会抬高对基本功的要求 —— 所以更要靠紧它们：

- **干净的接缝** —— [拆解](/zh/guides/planning)到各部分真正独立
- **版本控制纪律** —— 每个 agent 一个分支；有意识地整合，而非碰运气
- **强验证** —— 你没法盯每个会话，所以每个合并前都必须由[测试](/zh/guides/verification)把关
- **一种跟踪状态的方式** —— 知道每个会话在做什么、进行到哪

## 长时运行的 agent

agent 越来越能连续工作很久 —— 跨许多步骤地规划、迭代、从失败中恢复。要安全地委派一个长任务：

- 给它一份清晰的 [spec](/zh/workflows/spec-first) 和一个你审查过的规划
- 定义**检查点**，让它在有分量的决策上暂停、等你判断
- 确保它有一个客观信号（测试、构建）可以在无人值守下自我纠正

## 别过度伸展

并行是有开销的：更多分支、更多整合、更多要审查。对单个小任务，一个聚焦的会话仍然更优。当工作**真正能拆**成独立部分时才动用多 agent —— 而不是为了显得忙。你不需要的协调，只是纯粹的成本。
