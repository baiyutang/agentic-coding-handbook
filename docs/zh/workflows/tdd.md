---
title: 测试驱动开发
description: 把规格变成一个失败的测试，再让 agent 使它通过。
---

测试驱动开发和 agentic coding 是天作之合。测试是一份明确、可执行的规格 —— 恰是 agent 能反复迭代、无需你为每一步当裁判的那种目标。如果你只从这本手册里采纳一个工作流，就选它。

## 为什么 TDD 和 agent 如此契合

- **测试定义了"完成"** —— 不必猜需求
- **agent 得到客观信号** —— 红或绿，毫不含糊
- **它能无人值守地迭代** —— 运行、读失败、修复、重复，直到全绿
- **它约束了范围** —— agent 改的是生产代码，不是测试

## 循环

```markdown
1. 写一个（或让 agent 写一个）描述你想要行为的失败测试
2. 确认它是因为正确的原因而失败
3. 让 agent 在不改测试的前提下使它通过
4. 变绿后审查 diff
5. 以测试为安全网进行重构
```

## 先写测试

对行为要具体，而非实现：

```markdown
✅ "写一个失败测试：applyDiscount(100, '10PERCENT') 返回 90，
   未知的 code 则原价返回。然后实现 applyDiscount 使其通过。"
```

接着，这一句很关键：

```markdown
"只通过修改实现来让测试通过。不要编辑测试文件。
运行测试套件并迭代，直到全绿。"
```

"不要改测试"这句很重要 —— 否则一个卡住的 agent 可能会靠削弱断言来"修复"失败。

## 防止钻空子

要盯住两种失败模式：

- **削弱测试** —— 删掉一个断言，或改掉期望值。在指令里把测试钉成只读，并检查 diff 只动了源码。
- **过拟合** —— 把测试用到的那一个输入硬编码进去。加上第二个用例，或要求一个必须普遍成立的性质。

## 让测试驱动重构

一旦变绿，重构就是安全的：agent 可以放手重组结构，而套件会在行为一旦改变的那一刻就告诉它。

```markdown
"测试通过了。现在把定价规则抽成一张查找表，简化 applyDiscount。
全程保持套件为绿。"
```

这是[验证](/zh/guides/verification)的最佳形态 —— agent 既改动又检查，而你审查的是一份已被证明正确的 diff。
