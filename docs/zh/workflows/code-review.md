---
title: 代码审查
description: 把 agent 当作首轮审查者，并高效审查 agent 自己的产出。
---

在 agentic coding 里，审查是双向的：agent 可以审查代码（你的或它自己的），你也要审查 agent 的产出。稍加结构化，两者都会更好。

## 审查 agent 的产出

agent 的[验证循环](/zh/guides/verification)已经处理了机械层面的正确性 —— 能编译、过测试、过类型检查。这让你的审查得以聚焦于机器会漏掉的东西：

- **意图** —— 这解决的是真问题，还是一个长得像的问题？
- **设计** —— 这是对的抽象吗？还是会和下一次改动打架？
- **范围** —— 它有没有"顺手"改了你没让它改的东西？
- **安全与数据** —— 鉴权检查、输入校验、密钥、PII 处理
- **删除与风险** —— 有没有删掉或覆盖了不该动的东西？

审查 **diff**，而不是那段描述。agent 的总结往往过于乐观，diff 才是事实基准。

## 让 diff 保持可审查

小而聚焦的改动可审查，蔓延失控的改动会藏 bug。如果一个改动大到你没法有把握地审查，那其实是个[规划](/zh/guides/planning)问题 —— 让 agent 拆分它。一次提交只做一个逻辑改动，才能让审查诚实。

## 把 agent 当审查者

agent 是一个绝佳的首轮审查者，从不疲惫、从不略读：

```markdown
"审查已暂存的 diff，找出 bug、缺失的错误处理和安全问题。
每条发现都注明文件和行号并解释风险。先不要改任何东西。"
```

要求它_先别急着修_，能让你掌握主动 —— 在任何代码改动之前，由你来判断哪些发现是真的。把输出当作清单，而非定论；agent 会误报，也会漏掉你脑子里才有的上下文。

## 用新鲜的眼光审查自己的 PR

在开 PR 之前，让 agent 把改动讲回给你听：

```markdown
"像写 PR 描述那样，总结这个分支改了什么、为什么。
标出任何看起来不是有意为之的地方。"
```

如果它的总结和你的意图对不上，那代码多半也对不上 —— 而你在人类审查者不得不介入之前就抓到了它。
