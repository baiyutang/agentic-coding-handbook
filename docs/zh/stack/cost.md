# 成本与效率

agentic 工具的计费方式各不相同 —— 固定订阅、按 token 计费的 API，或两者混合。但无论哪种，控制成本的那些习惯，同时也会让 agent **更快、更准**，因为它们最终都指向同一件事：**保持上下文精简**。

## 你实际在为什么付费

在按 token 计费下，成本随**每一轮处理的上下文量**增长。每条消息都会重新处理到目前为止的对话。这意味着：

- 一个冗长杂乱的会话，**每一轮**都比短而聚焦的会话更贵 —— 即便问的是同一个问题。
- 把整个代码库倒进上下文既昂贵，又会稀释信号。
- 成本和质量通常同向移动：更精简的上下文既更便宜，**也**更锐利。

## 既省钱又提质的杠杆

- **给模型选对尺寸。** 机械、高频的工作用[快速层](/zh/stack/models)，把顶层留给推理。这是单个最大的杠杆。
- **让会话保持聚焦。** 一个任务一个会话；无关任务之间 `/clear`，会话变长时 `/compact`。见[上下文管理](/zh/guides/context-management)。
- **只拉入刚好够用的。** 把 agent 指向它需要的具体文件，而不是整个仓库。
- **投资 CLAUDE.md。** 把持久事实放进 [CLAUDE.md](/zh/reference/claude-md)，就不必每个会话重新解释你的技术栈。
- **先规划。** 一份[经过审查的规划](/zh/guides/planning)能避免昂贵的错误方向 —— 重写 400 行代码，远比花 30 秒重新规划贵。

## 什么时候该多花

省钱不是目的，**注意力的回报**才是。以下情况多花是值得的：

- 任务偏重推理，一次做对能省下好几个小时
- 你在规划或调试某个微妙的问题 —— 更好的方案会在下游产生复利
- 替代方案是消耗**你自己**的时间，而这几乎总是最贵的资源

要记住的框架：一个工程师小时的成本，远远压过哪怕是重度的模型使用。**在浪费的地方优化 agent 的花费，而不是在它真正给你带来杠杆的地方。**

## 订阅 vs. API

- **订阅（固定费率）** —— 成本可预测，适合稳定的个人使用；你不必再纠结每 token 的花费。
- **API（按量付费）** —— 随使用量扩展，更适合自动化、CI 和团队；让模型选择成为直接的成本杠杆。

按你使用量的波动程度来选。具体套餐请查各工具的**最新页面** —— 这正是经常变动的那类细节。
