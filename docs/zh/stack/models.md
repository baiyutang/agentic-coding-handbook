# 模型选择与成本

大多数 agentic 工具都允许你选用哪个模型运行。"永远用最大的模型"这个本能既贵，又常常慢得毫无收益。真正的技巧，是让模型匹配任务——同时搞清楚这个选择在成本上意味着什么。

想要**带基准数据的跨厂商对比**，见[模型对比](/zh/stack/models-compared)。

## 分层

模型家族通常分几个层级，在**能力**与**速度、成本**之间权衡。以 Claude 为例：

| 层级 | 特点 | 什么时候用 |
|---|---|---|
| **Opus** | 能力最强；推理最深 | 疑难调试、架构设计、棘手重构、需求模糊 |
| **Sonnet** | 均衡；强力默认 | 大多数日常编码、功能开发、代码审查 |
| **Haiku** | 最快、最便宜 | 高频、机械、或已明确定义的任务；快速小改；紧凑循环 |

这个形态在各家厂商都成立 —— 一个前沿层、一个均衡层、一个快速层。名字和版本号常变，**分层原则不变**。

## 核心原则

> 在**思考**是瓶颈的地方用最强的模型；在**打字**是瓶颈的地方换更快的模型。

- **受推理约束的工作** —— 规划功能、诊断微妙的 bug、设计抽象、从模糊需求出发 → **顶层**。方案的质量决定了下游的一切，值得这份花费。
- **机械性工作** —— 把一个已明确定义的改动铺到多个文件、格式化、样板代码、简单测试脚手架 → **快速层**。更强的模型很少做得更好，只是更慢更贵。

## 任务中途切换模型

你不必为整个会话锁定一个模型。一个高杠杆的常见套路：

1. 用**顶层**模型**规划**（把方案定对）
2. 用**更快**的模型**执行**机械步骤
3. 只在卡住时**升级**回强模型

在 Claude Code 里，会话中用 `/model` 命令切换，或启动时用 `--model`。

## 别过度优化

如果拿不准，**均衡层就是编码的正确默认** —— 它能很好地处理绝大多数任务。遇到真正的困难再往顶层够，工作明显机械时再往快速层降。在每一条提示上纠结模型选择不值得占用注意力，把这份刻意留给两个极端。

## 你实际在为什么付费

在按 token 计费下，成本随**每一轮处理的上下文量**增长。每条消息都会重新处理到目前为止的对话。这意味着：

- 一个冗长杂乱的会话，**每一轮**都比短而聚焦的会话更贵 —— 即便问的是同一个问题。
- 把整个代码库倒进上下文既昂贵，又会稀释信号。
- 成本和质量通常同向移动：更精简的上下文既更便宜，**也**更锐利。

## 既省钱又提质的杠杆

- **给模型选对尺寸。** 上面的层级选择是单个最大的杠杆 —— 机械高频工作换用快速层，成本可降 10–20 倍，且不损失质量。
- **让会话保持聚焦。** 一个任务一个会话；无关任务之间 `/clear`，会话变长时 `/compact`。见[上下文管理](/zh/guides/context-management)。
- **只拉入刚好够用的。** 把 agent 指向它需要的具体文件，而不是整个仓库。
- **投资 CLAUDE.md。** 把持久事实放进 [CLAUDE.md](/zh/reference/claude-md)，就不必每个会话重新解释你的技术栈。
- **先规划。** 一份[经过审查的规划](/zh/guides/planning)能避免昂贵的错误方向 —— 重写 400 行代码，远比花 30 秒重新规划贵。

## 什么时候该多花

省钱不是目的，**注意力的回报**才是。以下情况多花是值得的：

- 任务偏重推理，一次做对能省下好几个小时
- 你在规划或调试某个微妙的问题 —— 更好的方案会在下游产生复利
- 替代方案是消耗**你自己**的时间，而这几乎总是最贵的资源

要记住的框架：一个工程师小时的成本，远远压过哪怕是重度的模型使用。**在浪费的地方优化 agent 的花费，而不是在它真正给你带来杠杆的地方。**

## 订阅 vs. API

- **订阅（固定费率）** —— 成本可预测，适合稳定的个人使用；你不必再纠结每 token 的花费。
- **API（按量付费）** —— 随使用量扩展，更适合自动化、CI 和团队；让模型选择成为直接的成本杠杆。

按你使用量的波动程度来选。具体套餐请查各工具的**最新页面** —— 这正是经常变动的那类细节。
