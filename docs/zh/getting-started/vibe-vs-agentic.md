# Vibe vs Agentic Coding

用 AI 写代码并非只有一种。业界浮现出两种模式，选错常常是痛苦的根源。重点不在于哪个更好 —— 而在于**每种模式适配项目成熟度的不同阶段**。

## 两种模式

**Vibe coding（凭感觉编码）** —— 你用自然语言描述想要什么，接受返回的结果，凭感觉迭代。你不逐行读代码，而是看它*能不能跑*来判断。快、顺、适合探索。

**Agentic coding** —— 你把代码库、工具和约束交给 agent，它来规划、实现、测试、迭代 —— 但你始终在环内：审查规划、读 diff、以测试把关。启动更慢，但在规模化下可靠得多。

| | Vibe coding | Agentic coding |
|---|---|---|
| **你优化的是** | 探索速度 | 规模化的可靠性 |
| **你审查的是** | 结果（"能跑吗？"） | 规划和 diff |
| **最适合** | 原型、演示、一次性脚本 | 任何有真人依赖的东西 |
| **失败模式** | 交付脆弱、未审查的代码 | 琐碎小事上更慢 |

## 一条经验法则

> 如果它坏了、会有人打电话给你，就用 **agentic coding**。

黑客松原型、个人脚本、为学某个 API 打的快速探针？尽管 vibe。支付流程、认证系统、任何有真实用户或真实数据的东西？走 agentic —— 规划、验证、审查 diff。

::: warning 一个前车之鉴
凭感觉写出来的产品曾带着**硬编码在客户端的密钥、且没有认证**就上线给真实用户 —— 几天内就被攻破。让它快速上线的那个模式，对一个真会被人攻击的东西来说是错的模式。当事情重要时，[安全](/zh/guides/security)和[验证](/zh/guides/verification)里的纪律不是可选项。
:::

## 该转向 Agentic 的信号

留意这些 —— 它们意味着这个项目已经"长大到"超出了 vibe coding：

- 你改一个地方，某个**不相关**的地方坏了
- 你不确定 AI 到底**改了什么**
- 你已经无法自信地解释某个功能是怎么工作的
- 代码要走向**真实**了 —— 涉及用户、金钱或数据
- 你需要它下个月还能跑，且由别人维护

当其中两三个出现时，就切换模式：慢下来，[写一份 spec](/zh/workflows/spec-first)、[规划](/zh/guides/planning)、并把[验证](/zh/guides/verification)搭好。这本手册的其余部分，讲的就是如何把 agentic coding 做好。
