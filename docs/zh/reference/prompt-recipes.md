# Prompt 配方库

针对你最常做的任务的可复用起手式。复制一个，填上方括号，再按需调整。每个都编码了[指南](/zh/guides/)里的一个实践 —— 目的是让"好模式"成为省事的默认。它们是起点，不是咒语；尽管改。

## 创建 Spec

```markdown
我想构建：[一句话描述]。

用以下板块起草一份 Intent Spec，然后在我们规划之前问我任何澄清问题：
- **Goal** —— 完成后什么会成立，以及为什么重要
- **Acceptance Criteria** —— 具体、可检验的陈述（每条都能映射到测试）
- **Constraints** —— 要遵循的模式、不要动的东西、要守的限制
- **Non-Goals** —— 本轮明确排除在范围之外的
```

## 制定实现规划

```markdown
在写任何代码之前，为 [任务] 产出一份规划：
- 你会改哪些文件，以及每处大致改什么
- 步骤顺序，依赖在前
- 每一步你会如何验证（哪个测试或检查）
- 任何需要我确认的假设或未知项

等我确认后再实现。
```

## 按 Spec 实现

```markdown
实现改动以满足 [SPEC.md / 上面的标准]。
- 遵循我们商定的规划
- 每一步之后，运行 `[测试命令]` 和 `[类型检查命令]`
- 修好所有失败项；两个都变绿之前不要停
- 不要超出 spec 扩大范围 —— 任何让你想扩的地方，标出来告诉我
```

## 先写失败测试（TDD）

```markdown
写一个失败测试，规定：[行为，带具体输入和期望输出]。
确认它因正确原因失败。然后只实现让它通过所需的源码 —— 不要改测试。
运行套件并迭代，直到全绿。
```

## 调试失败

```markdown
这是一个失败测试和堆栈：[粘贴]。
在改任何代码之前，解释根因并指出负责的具体行号。
然后提出修法、应用它、跑完整套件确认没有别处回归。不要改测试。
```

## 审查 diff

```markdown
审查已暂存的 diff，找出 bug、缺失的错误处理和安全问题
（认证、输入校验、密钥、注入）。
每条发现都注明文件和行号并解释风险。先不要改任何东西 —— 先给我清单。
```

## 安全重构

```markdown
这是对 [目标] 的纯重构 —— 没有行为改动。
分成一个个都能保持构建为绿的小切片；每个切片后跑套件。
如果某个测试需要改动才能通过，停下来告诉我，而不是去改它。
```

## 讲解这个代码库

```markdown
我是这个代码库的新人。给我一张地图：主要模块及各自作用、数据如何流经 [功能]、
以及我要改 [X] 该从哪下手。用路径指出关键文件。
```

## 为生产加固

```markdown
审查 [功能] 的生产就绪度：认证与访问控制、输入校验、密钥处理、错误处理、边界情况。
列出具体缺口，带文件/行号，每个给一个建议修法。先不要动手改。
```

::: tip 提示
把在你项目里好用的配方存进 [`CLAUDE.md`](/zh/reference/claude-md) 或团队片段文件 —— 最好的 prompt 库，是为你自己代码库调过的那个。
:::
